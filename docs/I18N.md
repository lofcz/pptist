# PPTist Internationalization (typesafe-i18n)

PPTist uses [typesafe-i18n](https://github.com/ivanhofer/typesafe-i18n) v5.27.1, aligned with [sciobot-next](https://github.com/sciobot/sciobot-next) conventions.

## Why `baseLocale: en`?

Sciobot-next uses `cs` as its base locale because Czech is the primary product language. PPTist is being **migrated from Chinese UI strings** to en/cs/sk/pl. During mass migration, **English is the canonical source for keys** — translator agents write English first, then cs/sk/pl counterparts. This matches the migration direction (Chinese → English → CEE languages) and keeps key naming stable for the 50 parallel agent batches.

Shared locale union with sciobot: `cs | en | sk | pl`.

## Locales and namespaces

| Locale | Folder |
|--------|--------|
| en (base) | `src/i18n/en/` |
| cs | `src/i18n/cs/` |
| sk | `src/i18n/sk/` |
| pl | `src/i18n/pl/` |

Namespaces (each has `index.ts`):

| Namespace | Purpose |
|-----------|---------|
| `common` | Shared buttons, errors, loading |
| `editor` | Editor chrome, panels, toolbars |
| `canvas` | Canvas operations, alignment, link dialog |
| `screen` | Presentation mode |
| `mobile` | Mobile UI |
| `export` | Export dialogs |
| `configs` | User-visible labels from configs (hotkeys, animations, charts, shapes, …) |
| `components` | Shared component strings |

Each locale folder has a root `index.ts` (empty `{}`) that re-exports nothing manually — typesafe-i18n discovers namespace subfolders automatically.

## Commands

```bash
npm run i18n          # watch mode during development

```

## Vue SFC usage

```vue
<script lang="ts" setup>
import { useI18nContext } from '@/i18n/useI18nContext'

const { LL, locale, setLocale } = useI18nContext()
</script>

<template>
  <button>{{ LL.common.save() }}</button>
  <span>{{ LL.editor.toolbar.undo() }}</span>
</template>
```

Pattern: `const { LL } = useI18nContext()` then `LL.<namespace>.<key>()` in template or script.  
`LL` is a Vue `Ref` — auto-unwrapped in templates; use `LL.value` in script if needed outside template.

> **Note:** The Vue adapter exports `typesafeI18n()` and `i18nPlugin` from `i18n-vue.ts` (not a React-style `TypesafeI18n` wrapper component). `useI18nContext.ts` is a thin alias matching sciobot-next naming.

## Plain TypeScript files (hooks, configs, stores)

```typescript
import { getLL } from '@/i18n/getLL'

const LL = getLL()
message.success(LL.common.save())
```

`getLL()` uses synchronous loading via `i18n-util.sync` — suitable for module-level config arrays and non-component code.

## Locale detection and host sync

`src/i18n/locale.ts`:

| Source | Priority |
|--------|----------|
| `?locale=cs` URL param | 1 |
| `window.__PPTIST_LOCALE__` | 2 |
| Default `en` | fallback |

```typescript
import { getPptistLocale, setPptistLocale } from '@/i18n/locale'
import type { Locales } from '@/i18n/locale'
```

### sciobot-next (React host) integration

When PPTist is embedded in sciobot-next:

1. **Mount prop / global before boot:** set `window.__PPTIST_LOCALE__ = 'cs'` before the PPTist bundle loads.
2. **URL param:** load iframe/src with `?locale=cs`.
3. **postMessage (future):** host sends `{ type: 'pptist:setLocale', locale: 'sk' }`; PPTist listener calls `setPptistLocale`, reloads namespaces via `loadNamespaceAsync`, then `setLocale` from `useI18nContext()`.

Shared locale union: `cs | en | sk | pl` — same as sciobot-next `Locales` type.

## Vue vs React adapters

| Project | Adapter | Provider |
|---------|---------|----------|
| sciobot-next | `react` | `<TypesafeI18n locale={locale}>` |
| PPTist | `vue` | `app.use(i18nPlugin, locale)` |

Same translation file shape (`src/i18n/<locale>/<namespace>/index.ts`). Separate adapters, identical namespace layout. A future monorepo could share locale folders between apps.

## Czech plurals

When adding pluralized strings, Czech requires **6 plural forms** in typesafe-i18n syntax:

```
{count} {{zero|one|two|few|many|other}}
```

See [typesafe-i18n pluralization docs](https://github.com/ivanhofer/typesafe-i18n#plural-rules).
